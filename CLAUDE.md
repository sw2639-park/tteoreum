# 떠오름 — 프로젝트 가이드 (영구 컨텍스트)

이 폴더에서 작업할 때 항상 이 문서를 기본 지침으로 따를 것.
사용자(세웅)는 비개발자이고, 이 작업을 위해 다른 채팅(Claude.ai 등)으로
돌아가지 않음. **결정이 필요하면 지금 이 자리(Claude Code)에서 직접 물어볼 것.**
"다른 곳에서 확인하라"는 식으로 사용자를 다른 도구로 보내지 말 것.
스펙에 답이 있는 건 다시 묻지 말고 그대로 따를 것.

**시작 지시**: 사용자가 "시작해" 또는 이와 비슷한 짧은 말만 해도, 이 문서를
전부 읽은 뒤 Phase 0부터 곧바로 작업을 시작할 것. 별도의 상세한 프롬프트를
기다리지 말 것 — 필요한 내용은 모두 이 문서와 `docs/tteoreum-build-spec.md`에
있음. 각 Phase의 "완료 체크리스트"를 스스로 검증하면서 멈추지 않고 다음
Phase로 자동 진행할 것(사용자 행동이 실제로 필요한 시점만 예외 — 아래
"막힐 때만 직접 물어볼 것" 참고).

---

## 무엇을 만드는가

GTD식 "캡처함" 개인용 PWA. 떠오른 생각을 최소 마찰로 적어두고,
나중에 "처리됐는지 아닌지"만 추적함. AI가 우선순위·구현가능성 같은 걸
자동 평가하는 기능은 의도적으로 제외함(맥락 없는 AI 점수는 노이즈라고 판단).

전체 배경·결정사항·데이터 스키마·코드 스켈레톤은 **`docs/tteoreum-build-spec.md`**
가 1차 출처임. 이 문서와 스펙이 충돌하면 스펙을 따르고, 둘 다 없는 디테일은
아래 "기술적 결정"과 "구현 디테일" 항목을 따를 것.

참고 모형(시각 참고용, 코드 그대로 베끼지 말 것):
- `docs/popup-input.html` — 팝업 입력 화면의 배경 흐림 + 카드 스타일
- `docs/entry-comparison.html` — 알림 탭 vs 아이콘 탭 진입 흐름 차이
- `docs/graph-view-prototype-space.html` — **그래프뷰의 비주얼·인터랙션 확정 기준**.
  이건 "참고용 느낌"이 아니라 실제 색상 계산 로직(구심점/연결/미연결)과
  D3 force simulation 구조를 그대로 가져다 써도 되는 수준의 기준 파일임.

## 기술 스택 (고정, 다른 선택 하지 말 것)

- 프레임워크 없음. 번들러 없음. 순수 HTML/CSS/JS(ES 모듈).
  → 비개발자가 나중에 부분 수정을 다시 Claude Code에 맡길 때
    빌드 체인이 있으면 깨지기 쉬워서 일부러 안 씀.
- 저장: 브라우저 내장 IndexedDB. 외부 라이브러리(idb 등) 쓰지 말고
  네이티브 API로 작은 wrapper 직접 작성.
- 배포: Vercel.
  - PWA 본체 → 정적 사이트, 프로젝트명 **`tteoreum`**
  - 중계 서버(Phase 1부터 점검 푸시, Phase 2부터 AI 키 사용) → 별도 프로젝트,
    **`tteoreum-relay`** (PWA 본체와 절대 같은 프로젝트로 합치지 말 것 — 스펙 §8.2)
- GitHub: 새 저장소 `tteoreum` 생성 후 Vercel과 연결해 push 시 자동배포.
  (git/gh CLI 사용 불가하면 Vercel CLI로 직접 배포. 그건 알아서 판단.)

### 프로젝트 구조 (가이드, 합리적이면 조정 가능)
```
/
  index.html
  manifest.json
  sw.js
  /js
    db.js       (IndexedDB wrapper)
    popup.js    (팝업 입력)
    inbox.js    (인박스 + 스와이프)
    detail.js   (상세 화면)
    app.js      (쿼리파라미터로 진입경로 분기)
  /css style.css
  /icons
  /api          (Phase 1부터, tteoreum-relay 쪽)
```

### 디자인 토큰 (그대로 사용)
```
/* 라이트 (기본) */
--bg: #F6F5F2;
--ink: #191919;
--muted: #9A968D;
--line: #E3E0D9;
--accent: #FF5A36;   (포인트 컬러, 절제해서 한 곳에만)
--idea: #2D5BFF;     (아이디어 칩)
--note: #9A968D;     (메모 칩)

/* 다크 — prefers-color-scheme: dark 일 때 자동 전환, 토글 없음 */
--bg: #1C1B19;
--ink: #F0EEE9;
--muted: #8C887F;
--line: #2C2A27;
--accent: #FF6B47;   (다크에서는 살짝 톤 낮춤, 눈부심 방지)

폰트: -apple-system, "Apple SD Gothic Neo", "Pretendard", sans-serif
```
웹폰트 새로 안 불러옴. 전체 톤은 미니멀·절제 — 그라데이션, 장식, 그림자 과용 금지.
다크모드는 CSS `@media (prefers-color-scheme: dark)`로만 처리. JS 토글, 설정 항목 없음.

**예외: 그래프뷰 화면은 항상 다크 고정** (시스템 라이트/다크 설정 무관).
우주 테마 — 깊은 남색 배경, 별빛 점, 네뷸라 블롭, 비네트. 색상은 장식이 아니라
관계 정보(구심점=진하게, 연결된 별=옅게, 미연결=무채색 회색), 포인트 컬러
골드(`#FFD27A`)는 포커스 시에만. 정확한 구현은 `docs/graph-view-prototype-space.html` 참고.

## 구현 디테일 (스펙에 명시 안 됐거나 모호한 부분)

**공유 타겟(Web Share Target) 구현 방식** — 정적 호스팅이라 POST를 받을 서버가 없음.
- manifest.json에 share_target 등록(스펙 §1)
- service worker의 `fetch` 이벤트에서 `/share-target` 경로 POST를 가로채
  `request.formData()`로 파싱
- 서비스워커 컨텍스트에서 바로 IndexedDB에 CaptureItem 저장
  (서비스워커도 같은 origin의 IndexedDB 접근 가능)
- 저장 후 index.html로 리다이렉트
- → 서버 함수 없이 서비스워커 안에서 전부 처리.

**팝업 입력 vs 인박스 분기**
- `index.html` 로드 시 `location.search` 확인
- `?popup=1` 있으면 → 인박스를 배경에 깔고 팝업 오버레이 바로 표시
- 없으면 → 인박스만
- 공유로 들어온 경우는 저장이 이미 끝났으니 팝업 없이 인박스로

**아이콘**
- 실제 디자인 자산 없음. 단순 도형(원이나 짧은 선 하나)으로
  192x192 / 512x512 / maskable 아이콘 + 상태바용 흑백 단색 실루엣
  (`status-mono.png`) 직접 생성. 화려하게 만들지 말 것 — 여백 많은 단색 마크 하나면 됨.

**빈 상태**
- 인박스에 항목 없을 때 문구 `쌓인 거 없음` + 가운데 "+" 버튼.

**분류 칩 (스펙 §7) — AI 아님**
- 팝업 입력 카드에 `아이디어`/`메모` 칩 2개. 기본 선택은 `메모`, 탭하면 `아이디어`로 전환.
- 자동분류 코드는 아예 작성하지 말 것 — 캡처 시 사용자가 직접 정하는 게 전부.
- 상세 화면에서도 같은 칩으로 나중에 재분류 가능.

**자동 태깅 + 그래프뷰 (스펙 §7.1, Phase 2부터)**
- 위 분류 칩과는 완전히 다른 시스템. 태그는 AI가 자동으로 붙임(앱 열 때 배치,
  `tags` 비어있는 항목만, 자유형 1~3개).
- 그래프뷰는 인박스와 별개의 보조 화면(🕸 아이콘으로 진입). 인박스를 그래프뷰로
  바꾸지 말 것 — 메인 화면은 끝까지 인박스 리스트로 유지.
- 노드=항목, 같은 태그 공유 시 선으로 연결. 태그 없는 항목은 고립 노드로
  정상 표시(버그 아님).
- **이 화면에서만** D3.js를 CDN으로 불러와 force-directed layout 사용 허용.
  다른 화면에는 어떤 외부 라이브러리도 추가하지 말 것 — 예외는 이 화면 하나뿐.

**휴지통 (스펙 §4.1)**
- 길게 누르기 폐기 = 즉시 영구삭제 아님. `status: 'discarded'` + `discardedAt`만
  기록하고 휴지통 화면으로 보냄(별도 화면, 인박스 우측 상단 🗑에서 진입).
- 복원 누르면 `status: 'unhandled'`, `discardedAt` 제거.
- 앱이 열릴 때마다(`app.js` 진입점에서) `discardedAt`이 7일 넘은 항목을
  찾아 IndexedDB row 자체를 삭제. 별도 서버/크론 없음 — 순수 클라이언트 체크.

**보류(스누즈) 모달**
- 프리셋 4개: 내일 / 3일 후 / 다음 주 / 직접 선택(날짜 피커).
  풀 캘린더를 기본으로 띄우지 말 것 — 프리셋이 우선.

**인박스 그룹핑**
- 날짜 구분선만: `오늘` / `이번 주` / `오래됨`. 접고 펼치는 기능 없음,
  그냥 리스트 중간에 들어가는 텍스트 라벨.

## 진행 방식 — 전 단계 자율 진행

매번 사용자에게 "다음 단계 진행해도 될지" 물어보지 말고, 각 Phase 끝나면
아래 체크리스트로 스스로 검증한 뒤 다음 Phase로 자동 진행할 것.
API 키 발급, Vercel/GitHub 권한처럼 사용자 행동이 실제로 필요한 것만
그 시점에 직접 물어볼 것.

### Phase 0 — 캡처 + 로컬 저장 (서버·AI 없음)
구현: 캡처 입구(홈 아이콘/공유/숏컷), IndexedDB 저장, 인박스 스와이프,
팝업 입력. 데이터 모델은 스펙 §5 그대로.

**완료 체크리스트:**
1. Vercel 배포 URL을 안드로이드 Chrome에서 열고 "홈 화면에 추가" 가능
2. 홈 아이콘 탭 → 인박스(빈 상태, "쌓인 거 없음") 표시
3. 아이콘 길게 누르기 → "새 기록" → 팝업이 배경 어둡게 깔고 뜨고,
   입력창에 키보드 바로 올라옴, `아이디어`/`메모` 칩이 보이고 기본은 `메모`
4. 텍스트 입력 후 팝업 바깥 탭 → 자동저장 + 닫힘
5. 인박스에서 방금 적은 항목이 미처리로 보임, 날짜 구분선(오늘/이번주/오래됨) 정상 표시
6. 오른쪽 스와이프(처리) / 왼쪽 스와이프(보류 프리셋 모달) /
   길게 누르기(폐기 → 휴지통行) 전부 동작
7. 휴지통 화면에서 폐기 항목 복원 가능, 7일 지난 항목은 자동 영구삭제 로직 동작
8. 다른 앱에서 텍스트·링크 공유 → "떠오름" 선택 → 인박스에 들어옴
9. 와이파이·데이터 끄고도 캡처·인박스 정상 동작(완전 로컬 확인)
10. 폰 시스템을 다크모드로 바꾸면 앱도 자동으로 다크 톤으로 전환됨

통과하면 다음 Phase로.

### Phase 1 — 중계 서버 + 점검 알림 (AI 키 불필요)
구현: `tteoreum-relay` Vercel 프로젝트(스펙 §8.2), 일일 점검 푸시(§2.1, 아침 07:30).
자동분류는 없음(§7) — AI 키는 이 단계에서 필요 없음. Vercel 배포·크론 설정만 하면 됨.

**완료 체크리스트:**
1. 매일 07:30 잠금화면에 `미처리 N건` 알림 옴(같은 tag로 조용히 갱신, 알림 안 쌓임)
2. 알림 탭 → 팝업 입력으로 바로 이동(인박스 아님)
3. 일요일엔 알림에 주간 요약 한 줄 추가됨

### Phase 2 — ✨발전 + 자동 태깅 + 그래프뷰 (여기서 AI 키 처음 필요)
구현: 상세 화면 "✨발전"(스펙 §6, §8.2 `/api/develop`), 자동 태깅(§7.1,
`/api/tag`), 그래프뷰 화면(§7.1, 인박스 🕸 아이콘 진입).
✨발전·태깅 모두 사용자가 누를 때 또는 앱 열 때 배치로만 호출 — 자동 실행 절대 금지.
이 단계에서 AI API 키가 처음 필요함 — 사용자에게 직접 물어보고 Vercel
환경변수로만 저장(클라이언트에 노출 금지).

**완료 체크리스트:**
1. 상세 화면에서 "✨발전" 누르면 AI 결과가 `aiSummary`에 채워짐
2. 앱 열 때 태그 없는 항목들이 배치로 자동 태깅됨(누르지 않아도 동작)
3. 인박스 🕸 아이콘 → 그래프뷰 진입, 같은 태그 항목끼리 선 연결
4. 태그 없는 항목은 그래프에서 고립된 점으로 보임(에러 없음)
5. 그래프뷰는 폰 시스템 설정과 무관하게 항상 다크(우주 테마)로 보임
6. 클러스터 내 연결이 가장 많은 노드(구심점)가 가장 진한 색·가장 큰 글로우로 보이고,
   같은 클러스터의 다른 노드는 옅은 색, 미연결 노드는 무채색 회색
7. 노드 탭 시 그 노드+직접 연결 노드만 골드로 빛나고 나머지는 어두워짐
8. 그래프 노드에서 상세 화면으로 이동 가능
9. 태깅 API 실패해도 앱이 멈추지 않고 해당 항목은 그냥 고립 노드로 남음

### Phase 3 — 주간 요약 고도화 + 내보내기
처리 시 캘린더/할일 앱으로 원클릭 내보내기(스펙 §2.2, §10).
**COS로 보내기**(스펙 §2.3) — 상세 화면에 버튼 하나만 추가. `idea` 타입 항목에만
노출. `navigator.share()`로 COS PWA(Web Share Target 등록되어 있음)에 원문
전달, 사용자가 공유 시트에서 COS를 탭하면 완료. 떠오름은 전송 시도 여부만
`sentToCos`로 기록하고, COS 쪽 저장 성공 여부는 확인하지 않음(두 앱 DB는
서로 모르는 상태 유지).

**완료 체크리스트:**
1. 아이디어 타입 항목 상세에서만 `COS로 보내기` 버튼 보임(메모 타입엔 없음)
2. 버튼 누르면 안드로이드 공유 시트가 뜨고 COS가 선택지에 보임
3. COS 선택 후 전송하면 떠오름 쪽 항목에 전송됨 표시(`sentToCos: true`)

## 절대 하지 말 것
- 아이디어/메모 분류 칩(§7)을 AI가 자동으로 정하게 만들기 — 이건 여전히 금지.
  (단, §7.1의 자동 태깅은 별개 시스템이라 Phase 2부터 허용됨 — 혼동하지 말 것)
- Phase 0 끝나기 전에 Phase 1 이후 기능(점검알림, ✨발전, 자동태깅, 그래프뷰, COS 연동) 코드 넣지 말기
- 인박스를 그래프뷰로 대체하기 — 그래프뷰는 항상 보조 화면
- 그래프뷰 외 다른 화면에 D3.js 등 외부 라이브러리 추가하기
- PWA 본체와 tteoreum-relay를 같은 Vercel 프로젝트로 합치기
- **떠오름과 COS(Chief of Staff, 별도 PWA)를 같은 코드베이스/저장소로 합치기**
  — 의도적으로 분리된 별개 앱임. 연결은 Web Share API 버튼 하나뿐(§2.3).
  COS의 내부 데이터 구조·API를 떠오름이 알거나 의존하게 만들지 말 것.
- 로그인/계정, 외부 analytics, 사용자 데이터를 서버 DB에 저장하는 것 — 전부 불필요
- 사용자를 Claude.ai 채팅 등 다른 곳으로 보내서 확인받게 하는 것

## 막힐 때만 직접 물어볼 것
- **스와이프 힌트 표시 방식** (스펙 §11 참고) — 항목이 가만히 있을 때도
  오른쪽=처리(초록)/왼쪽=보류(주황) 색깔이 가장자리에 살짝 보이게 할지,
  아니면 평소엔 안 보이고 실제로 밀 때만 나타나게 할지. Phase 0 인박스
  만들기 전에 먼저 물어볼 것.
- 위 내용으로 답이 안 나오는 진짜 기술적 충돌(권한, 플랫폼 제약 등)
- Phase 2 진입 시 AI API 키
- 그 외 작은 판단(파일 분리 방식, 변수명 등)은 합리적으로 정해서 진행하고
  멈추지 말 것. 가정한 내용은 각 Phase 완료 시 `README.md`에 "가정 목록"으로
  남길 것.
